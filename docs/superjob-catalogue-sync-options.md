# Варианты синхронизации профессиональной сферы (SuperJob ↔ форма редактирования)

Цель: чтобы выбранная в нашей форме «Отрасль / Специализация» при обновлении вакансии корректно отправлялась на SuperJob (поле `catalogues`), и при открытии формы по вакансии с SuperJob отображались те же значения.

В SuperJob:
- **Каталог** — дерево: отрасли (верхний уровень, `key`, без `id_parent`) и категории/позиции (`positions[]` с `key`, `id_parent`).
- В вакансию **передавать можно только категории** (элементы с `id_parent`), не отрасли. То есть в `catalogues` уходят id из `positions` (поле `key`).
- Наш бэкенд: `GET /api/superjob/catalogues` отдаёт каталог SuperJob (прокси к их API).

Сейчас в форме везде используется справочник HH (`/api/specializations`), его id не совпадают с `key` каталога SuperJob, поэтому подставлять их в `catalogues` нельзя.

---

## Вариант 1. Отдельный справочник SuperJob только при редактировании вакансии с SuperJob (рекомендуется)

**Идея:** Когда открыта форма редактирования вакансии, у которой есть публикация на SuperJob (platform_id === 4), подгружать каталог SuperJob и показывать в полях «Отрасль компании» / «Специализация» именно его (иерархия: отрасль → позиции). При сохранении отправлять в SuperJob выбранный id категории из этого каталога.

**Плюсы:**
- Полная синхронизация с SuperJob при редактировании вакансии с этой платформой.
- Не трогаем создание новых вакансий и другие платформы (HH, Avito, Rabota).
- Один источник правды для SuperJob — их же каталог.

**Минусы:**
- В режиме редактирования SuperJob-вакансии форма «переключается» на другой справочник (отрасль/специализация из SuperJob, а не из HH).
- Нужно при открытии формы определять, что это «редактирование вакансии с SuperJob», и подставлять в форму данные из текущей вакансии SuperJob (например из `existingVacancy.catalogues`).

**Реализация (кратко):**
1. В `AddPublication.vue`: если `props.editingVacancy?.platforms_data?.[0]` имеет platform id = 4 (SuperJob), вызывать `loadDictionaries('superjob')`.
2. В `loadDictionaries('superjob')`: запрос `GET /api/superjob/catalogues`, привести ответ к формату «отрасли + roles» (как у HH): каждый элемент верхнего уровня — отрасль с `key` как id, в `roles` — массив `positions` с `key` и `title`.
3. Хранить в форме выбранную **категорию** SuperJob (id = `position.key`), например в `data.superjob_catalogue_id` или в том же `data.industry` с полем `superjobCatalogueId` при платформе SuperJob.
4. При загрузке вакансии: если есть `existingSuperjobVacancy.catalogues[0].id`, подставлять его в форму (найти отрасль по id_parent, позицию по id и выставить оба выпадающих).
5. В `mapVacancyToSuperjobPayload`: если в formData есть `superjob_catalogue_id` (или выбранная категория из справочника SuperJob), передавать `payload.catalogues = [{ id: superjob_catalogue_id }]`; иначе — как сейчас из `existing.catalogues`.

## Вариант 2. Отдельный блок «Профессиональная сфера для SuperJob» (без замены отрасли/специализации)

**Идея:** Оставить текущие поля «Отрасль компании» и «Специализация» на справочнике HH. Добавить отдельный блок (или одно поле), который показывается только при редактировании вакансии с SuperJob: выпадающий список «Профессиональная сфера (SuperJob)», заполненный из `GET /api/superjob/catalogues` (например, плоский список всех позиций или двухуровневый выбор). При сохранении на SuperJob отправлять только это значение.

**Плюсы:**
- Не меняем логику существующих полей industry/specializations (они по-прежнему для БД и для HH/Avito/Rabota).
- Явно видно, что это «значение для SuperJob».

**Минусы:**
- Дублирование смысла (два способа задать «сферу»), возможная путаница у пользователя.
- Нужно хранить и отображать отдельное поле (например в `platforms_data` или в отдельном поле вакансии) для SuperJob catalogue id.

**Реализация (кратко):**
1. Добавить в форму поле (или два: отрасль SuperJob → специализация SuperJob), видимое при `editingVacancy && platformId === 4`.
2. Загружать `GET /api/superjob/catalogues`, показывать выбор; сохранять выбранный id категории в `data.superjob_catalogue_id`.
3. При открытии формы подставлять значение из `existingSuperjobVacancy.catalogues[0].id`.
4. В `mapVacancyToSuperjobPayload` использовать `formData.superjob_catalogue_id` для `payload.catalogues`.

## Вариант 3. Маппинг HH ↔ SuperJob по названиям

**Идея:** Справочники HH и SuperJob не менять. При сохранении вакансии на SuperJob по выбранным на сайте `industry` / `professional_role` (названия или id HH) искать наиболее подходящую категорию в каталоге SuperJob (по совпадению или похожести названий) и подставлять её id в `catalogues`.

**Плюсы:**
- Один общий вид формы для всех платформ, без переключения справочников.

**Минусы:**
- Неточности и расхождения: одному названию HH могут соответствовать несколько категорий SuperJob или ни одной.
- Нужно поддерживать и обновлять правила/таблицу маппинга.
- При импорте с SuperJob мы получаем только строки (industry/specializations); при следующем обновлении маппинг «строка → id SuperJob» может быть неоднозначным.

**Реализация (кратко):**
1. На бэкенде или во фронте хранить маппинг «id или name HH (отрасль/роль) → id категории SuperJob» (или «название → id»).
2. В `mapVacancyToSuperjobPayload` по `formData.industry` и `formData.professional_roles[0]` вычислять SuperJob catalogue id и подставлять в `payload.catalogues`.
3. При отсутствии соответствия — оставлять `existing.catalogues` или не менять сферу на SuperJob.

## Вариант 4. Универсальный справочник «по платформе» при выборе платформы

**Идея:** При выборе платформы публикации (HH, Avito, Rabota, SuperJob) всегда подгружать справочник этой платформы для полей «Отрасль / Специализация». Для SuperJob — `GET /api/superjob/catalogues`, для HH — текущий `getRolesHh()` и т.д. В форме хранятся id выбранной платформы; при обновлении на конкретную платформу отправляются только они.

**Плюсы:**
- Консистентность: что видишь в форме для платформы, то и уходит на эту платформу.
- Подходит, если форма всегда привязана к одной выбранной платформе (создание публикации на конкретную площадку).

**Минусы:**
- При редактировании «базовой» вакансии с несколькими платформами неочевидно, какой справочник показывать; возможны конфликты (вакансия и с HH, и с SuperJob).
- Требует явного выбора «редактируем под какую платформу» или разделения сценариев (создание под платформу X / редактирование вакансии с платформой Y).

**Реализация (кратко):**
1. В форме всегда определять «текущую платформу» (выбор пользователя или из `editingVacancy.platforms_data`).
2. `loadDictionaries(platform)` для `platform === 'superjob'` загружать каталог SuperJob и формировать `currectRole` в том же формате, что и для HH (отрасли с `roles` = позиции).
3. Хранить в данных формы id выбранной категории для текущей платформы (для SuperJob — id из каталога).
4. При сохранении на SuperJob использовать этот id в `mapVacancyToSuperjobPayload`.

---

## Рекомендация

Оптимально начать с **варианта 1**: при редактировании вакансии с SuperJob подгружать каталог SuperJob и подставлять в те же поля «Отрасль / Специализация» иерархию отрасль → позиции, а при сохранении отправлять выбранный id категории в `catalogues`. Так профессиональная сфера на SuperJob и в форме будут синхронизированы без дублирования полей и без неточного маппинга по названиям.

Если нужно, могу расписать конкретные правки по файлам (AddPublication.vue, loadDictionaries, mapVacancyToSuperjob, получение текущей вакансии SuperJob при открытии формы) под выбранный вариант.
