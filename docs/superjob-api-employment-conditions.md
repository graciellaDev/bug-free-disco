# SuperJob API: несколько значений для условий занятости

## Что говорит публичная документация API (api.superjob.ru, версия 2.0)

### Один объект на вакансию

В ответе по вакансии (GET) и в описании полей вакансии указано:

- **type_of_work** — один **объект** `{ id, title }`. Список значений из справочника `GET /2.0/references/`: `type_of_work`: 0, 6, 7, 9, 10, 12, 13. В этом же справочнике «график» по сути объединён с типом занятости (например, 12 — «сменный график работы», 7 — «временная работа / freelance»).
- **place_of_work** — один **объект** `{ id, title }`. Значения: 0, 1, 2, 3 (на территории работодателя, на дому, разъездной).
- **schedule** — в **официальной таблице полей вакансии** (описание ответа GET вакансии) поле **schedule не перечислено**. То есть в публичной документации для соискателя отдельного поля «график» нет — варианты графика входят в type_of_work. В нашем коде мы тем не менее читаем `publication.schedule` при импорте и передаём `payload.schedule` при обновлении (id из текущей вакансии), то есть реальный ответ API работодателя может содержать `schedule`.
- **is_substitute_work** — в сохранённой копии документации API **не встречается**. Возможно, это параметр API для работодателя (вакансия на замещение / временная работа). Имеет смысл проверить актуальную документацию (api.superjob.ru/doc/, раздел для работодателя api.superjob.ru/client/) и реальный ответ GET вакансии: если в ответе приходит **is_substitute_work** (например, bool), то при обновлении вакансии, скорее всего, нужно передавать именно его для признака «работа на замещение», а не полагаться только на **schedule**.

То есть в документации для **одной вакансии** тип занятости и место работы задаются **одним значением**, а не массивом.

### Справочник type_of_work (references)

```text
"type_of_work": {
  "0": "не имеет значения",
  "6": "полный рабочий день",
  "7": "временная работа / freelance",
  "9": "работа вахтовым методом",
  "10": "неполный рабочий день",
  "12": "сменный график работы",
  "13": "частичная занятость"
}
```

Значения с id 1, 2, 3 относятся к **place_of_work** (офис, удалёнка, разъездная). В наших константах `SUPERJOB_EMPLOYMENT_CONDITIONS` часть пунктов — это type_of_work (6, 10, 7, 9, 12, 13), часть — place_of_work (1, 2, 3). Значение 14 («Гибрид») в публичном справочнике не приведено — возможно, добавлено позже или в API для работодателя.

### Где допускаются массивы

- **catalogues** — в поиске можно передавать несколько id (commalist/array).
- **driving_licence** — массив категорий прав.
- **languages** — массив пар язык/уровень.

Для полей типа занятости/места работы в описании вакансии везде фигурирует один объект.

## Варианты поддержки «нескольких значений» у нас

### 1. Хранить несколько у себя, в API отправлять один

- В форме и в БД хранить массив выбранных условий (`superjob_employment_conditions`: массив id).
- При синхронизации с SuperJob (PUT вакансии) передавать в API **одно** значение:
  - **typeOfWork** — например, первый выбранный id из «типа занятости» (6, 7, 9, 10, 12, 13) или приоритетный.
  - **place_of_work** — первый выбранный из «места работы» (1, 2, 3) или приоритетный.
- Остальные выбранные пункты можно:
  - сохранять только у нас и показывать в карточке/форме, и/или
  - дублировать текстом в описании вакансии (work/compensation), чтобы они были видны соискателям.

Так мы не нарушаем текущий контракт API (одно значение на поле).

### 2. Уточнить API для работодателя

- Документация для соискателя: https://api.superjob.ru/
- Отдельно есть раздел для работодателя: https://api.superjob.ru/client/
- Имеет смысл проверить актуальную доку и примеры запросов **создания/обновления вакансии** (POST/PUT) на предмет:
  - параметров вида `type_of_work[]`, `schedule[]` или массивов в JSON;
  - полей вроде `employment_conditions` или `extended_search_parameters` при создании/редактировании вакансии.

Если там разрешён массив — можно будет отправлять все выбранные id.

### 3. Поле extended_search_parameters в ответе вакансии

В ответе вакансии есть поле **extended_search_parameters** (array). Назначение в документации расписано не полностью. Имеет смысл при открытии вакансии с SuperJob посмотреть в ответе API, приходят ли там несколько type_of_work/place_of_work/schedule или дополнительные ключи условий занятости — и при необходимости маппить их в наш массив `superjob_employment_conditions` при загрузке формы.

## Реализация в проекте

### Сохранение

1. **В нашей БД**  
   При обновлении вакансии через API (`mapVacancyToUpdateFormat`) в payload передаётся поле **`superjob_employment_conditions`** — массив строк (id выбранных тегов). Бэкенд должен принять и сохранить это поле (например, колонка JSON/array или отдельная таблица) и при GET вакансии возвращать его в ответе.

2. **В API SuperJob (PUT вакансии)**  
   В `mapVacancyToSuperjobPayload` массив `superjob_employment_conditions` используется так:
   - **typeOfWork** — первый выбранный id из «типа занятости»: 6, 7, 9, 10, 12, 13, 14;
   - **place_of_work** — первый выбранный id из «места работы»: 1, 2, 3.
   Остальные id (например `from_4h_day`, `weekends`, `evening`, `one_time`, `side_job`) в API SuperJob не отправляются и хранятся только у нас.

3. **При открытии формы**  
   В `fillFormFromCurrentVacancy` из вакансии (ответ нашего API) читается `vacancy.superjob_employment_conditions` и подставляется в `data.superjob_employment_conditions`, чтобы теги отображались выбранными.

### График: schedule и is_substitute_work

- Сейчас в коде при обновлении вакансии в payload передаётся **schedule** (id из `existing.schedule.id`). В официальной таблице полей вакансии поле schedule не описано; график в публичном API фактически входит в **type_of_work**.
- Если в ответе GET вакансии (или в документации API для работодателя) появится **is_substitute_work** (вакансия на замещение), стоит добавить его в интерфейс `SuperjobExistingVacancy`, при формировании payload передавать в API и при загрузке формы учитывать для отображения. Проверку можно сделать по реальному ответу `GET /api/superjob/vacancies/:id` (или эквиваленту на бэкенде).
- Итог: актуальным для «графика» / временной работы может быть **is_substitute_work**, а не только **schedule**. Рекомендуется свериться с актуальной документацией и ответом API.

### Рекомендация по бэкенду

- Добавить приём и сохранение поля **`superjob_employment_conditions`** (массив строк) при создании/обновлении вакансии.
- При отдаче вакансии (GET) возвращать это поле, чтобы форма редактирования подставляла сохранённые условия занятости.
